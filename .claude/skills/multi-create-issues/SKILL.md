# 一括Issue作成ワークフロー

複数の要望・バグ報告を受け取り、並列エージェントでコードベースを調査し、構造化されたIssueを一括作成する。

## 前提

- 引数 `$ARGUMENTS` にCSVファイルパスまたは要望リストのテキストが渡される
- CSVの場合: 各行が1つの要望
- テキストの場合: 改行区切りまたは番号付きリストで複数の要望が記述されている
- 引数がない場合はユーザーに要望リストの入力方法を確認する

## 手順

### 1. 要望リストの解析

入力を解析し、個別の要望に分割する。

**CSVファイルの場合:**
- Read ツールでCSVを読み込む
- ヘッダー行から列の意味を把握する
- 各行を1つの要望として抽出する

**テキストの場合:**
- 番号付きリスト、改行区切り、またはマークダウンリストから個別の要望を抽出する

各要望について以下の情報を整理する:
- **ID/番号**: 元リストの通し番号（あれば）
- **タイトル**: 要望の概要
- **詳細**: 補足説明（あれば）

### 2. 要望一覧のユーザー確認

解析した要望一覧をテーブル形式でユーザーに提示し、確認を取る:

```
| # | タイトル |
|---|---------|
| 1 | XXXの画面でYYYが... |
| 2 | ZZZのロジックが... |

上記N件のIssueを作成します。除外したい項目や修正があれば教えてください。
```

ユーザーが除外や修正を指示した場合は反映する。

### 3. 並列調査フェーズ

各要望について **Task ツール（Explore エージェント）** を並列起動し、コードベースを調査する。

各エージェントには以下を伝える:
- 要望のタイトルと詳細
- 調査項目: 関連ファイルのパス、問題箇所の特定、修正方針の提案、影響範囲
- 作業ディレクトリの絶対パス

**注意:** 並列エージェントは最大4-5件ずつ起動する。要望が多い場合はバッチに分けて実行する。

### 4. 調査結果の集約

各エージェントの調査結果を集約し、以下を確認する:
- 複数の要望が同じファイルに関わる場合は依存関係をメモする
- 調査で不明点があった要望はフラグを立てる

### 5. Issue一括作成

各要望について `gh issue create` でIssueを作成する。

**Issueタイトル形式:**
- 元リストに番号がある場合: 番号を含める（例: `[#XXX] 要望タイトル`）
- 番号がない場合: 要望タイトルのみ

**Issue本文テンプレート:**

```markdown
## 概要

[要望・バグの説明]

## 現状

[調査で特定した現在のコードの該当箇所]

\`\`\`
// ファイルパス:行番号
該当コードのスニペット
\`\`\`

## 修正方針

1. ...
2. ...

## 対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `ファイルパス` | 変更の概要 |

## テスト観点

- ...
```

### 6. サマリー報告

全件の作成結果をユーザーに報告する:

```
## 一括Issue作成サマリー

合計 N 件のIssueを作成しました。

| # | Issue | タイトル | 質問 | ステータス |
|---|-------|---------|------|-----------|
| 1 | #XXX | ... |      | 作成完了 |
| 2 | #YYY | ... | あり | 作成完了 |

### 質問事項の詳細
- #YYY: [質問内容]

### 依存関係のあるIssue
- #XXX と #YYY は同じファイルを変更するため、実装順序に注意

### 次のステップ
各Issueの内容を確認し、実装に進む場合は `/implement-issue <Issue番号>` を使用してください。
```
