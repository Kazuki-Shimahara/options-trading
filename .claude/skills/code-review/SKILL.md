# PRコードレビューワークフロー

指定されたPRの差分を多角的に分析し、インラインコメント付きのGitHubレビューを投稿する。

## 前提

- 引数 `$ARGUMENTS` にPR番号またはPR URLが渡される
- PR URLの場合は末尾の番号を抽出してPR番号として使用する
- 引数がない場合は現在のブランチに紐づくPRを `gh pr view --json number` で特定する
- `$ARGUMENTS` に `--self-review` フラグが含まれている場合、セルフレビューモードを有効にする
  - フラグはPR番号の前後どちらでもよい（`123 --self-review` / `--self-review 123`）
  - `$ARGUMENTS` から `--self-review` を除去した残りをPR番号/URLとして扱う

## 手順

### 1. PR情報の取得

```bash
gh pr view <PR番号> --json title,body,headRefName,baseRefName,author,number
```

owner/repo は `gh repo view --json nameWithOwner` で取得する。

### 2. 関連Issueの確認

- PR本文から `Closes #123` / `Fixes #123` / `Related Issue` パターンを抽出する
- 見つかれば `gh issue view <Issue番号> --json title,body,comments` で要件を把握する
- コメント内の追加要件も確認する
- Issue要件は後続のレビューで「要件を満たしているか」の観点として追加する

### 3. プロジェクト規約の確認

以下のファイルが存在すれば **必ず** 読み込む（存在しなければスキップ）:

- **CLAUDE.md**: コーディングスタイル・テスト規約・フォーマッター・ディレクトリ構成ルール
- **README.md**: プロジェクト概要・アーキテクチャ・コントリビューションガイドライン

これらの情報は後続のレビュー観点（特に規約準拠・設計）に組み込む。

### 4. 差分の取得と解析

```bash
gh pr diff <PR番号>
gh api repos/{owner}/{repo}/pulls/{PR番号}/files
```

- unified diff と変更ファイル一覧を取得する
- 各ハンクの行番号範囲を `@@ -a,b +c,d @@` から抽出する

**スキップ対象:**

- ロックファイル（`package-lock.json`, `yarn.lock`, `Gemfile.lock`, `poetry.lock`, `go.sum` 等）
- 自動生成ファイル（`generated`, `auto-generated` コメント付き）
- バイナリファイル

### 5. 変更ファイルの深堀り調査

各変更ファイルについて:

- **本体**: Read ツールで全体を読み、変更箇所の前後コンテキストを把握する
- **依存先**: インポート先のモジュール・型定義・インターフェースを読む
- **テスト**: 対応するテストファイルを Glob で探して読む（`*_test.*`, `*.test.*`, `*.spec.*`）。存在しなければ記録する
- **大規模PR対策**: 10ファイル超の場合はビジネスロジック → 変更行数多い順で優先する

### 6. 多角的レビュー（6観点）

以下の6つの観点で変更内容をレビューする:

| 観点 | チェック内容 |
|------|-------------|
| 正確性 | ロジックエラー、エッジケース、null/undefined、型不整合、エラーハンドリング |
| セキュリティ | injection、XSS、認証/認可漏れ、シークレット漏洩、入力バリデーション |
| パフォーマンス | N+1クエリ、不要ループ、メモリリーク、大量データ一括読み込み |
| テスト | 変更ロジックのテスト有無、エッジケーステスト、テスト品質。**ビジネスロジック・API・DB操作の変更に対応するテストが存在しない場合は must-fix として警告する** |
| 規約準拠 | CLAUDE.md / README.md 規約との整合性、命名規則の一貫性、既存パターンとの整合 |
| 設計 | 関心の分離、DRY、適切な抽象化、拡張性・保守性 |

### 7. セルフレビュー（`--self-review` 指定時のみ）

`--self-review` フラグが指定されていない場合はこのステップをスキップし、ステップ8へ進む。

ステップ6で作成した指摘リストを投稿前に再検証し、精度を向上させる。

#### 7a. コード再検証

各指摘の参照行を Read ツールで再度読み込み、指摘内容が実際のコードと一致するか確認する。

#### 7b. 偽陽性チェック

以下の観点で、既に対処済みの問題を誤検出していないか検証する:

- 上流のバリデーションで既に保護されている
- フレームワークが自動的に処理している
- 別ファイルのテストで既にカバーされている
- 呼び出し元で事前条件が保証されている

#### 7c. 行番号検証

インラインコメントの `line` がステップ4で取得した差分ハンク範囲内に含まれるか確認する。差分外の行を指している指摘はインラインコメントからサマリーへ移動する。

#### 7d. 重大度再評価

- 根拠が薄い must-fix を suggestion に降格する
- 些末な suggestion を nit に降格する

#### 7e. 重複・冗長チェック

- 同一根本原因に起因する複数の指摘を統合する
- 同一パターンの指摘は最初の1件のみインラインに残し、残りはサマリーに集約する

#### 過剰フィルタリング防止ルール

- 指摘を除外・降格するのは **具体的なコード根拠がある場合のみ** とする（推測で除外しない）

#### セルフレビュー結果の表示

セルフレビューの結果はユーザーへの報告にのみ含め、**GitHubレビューには投稿しない**。

以下を記録する:
- 除外した指摘の件数と理由
- 降格した指摘の件数と理由（変更前後の重大度）
- 統合した指摘の件数
- 行番号を修正した指摘の件数
- Before/After の指摘件数比較（重大度別）

### 8. 指摘整理 & GitHubレビュー投稿

#### 重大度分類

指摘を以下の3段階に分類する:

- 🚨 **must-fix**: バグ、セキュリティ、データ損失リスク
- 💡 **suggestion**: 改善推奨だが動作問題なし
- 🔍 **nit**: 軽微なスタイル指摘（最大3件まで、超過はサマリーに集約）

#### ノイズ低減ルール

- フォーマッターで直せるスタイル指摘はしない
- 同一パターンは最初の1箇所のみインライン、残りはサマリーで「同様N箇所」と記載する
- 明らかに意図的な設計判断への代替案提示は控える

#### イベントタイプ決定

- `REQUEST_CHANGES`: must-fix が1件以上
- `APPROVE`: must-fix 0件、suggestion 0-1件、全体的に品質良好
- `COMMENT`: must-fix 0件だが suggestion 複数、または判断保留

#### GitHub API レビュー投稿

`gh api` で1回のAPIコールでレビューを投稿する。インラインコメントとサマリーを同時に送信できる。

```bash
gh api repos/{owner}/{repo}/pulls/{PR番号}/reviews \
  --method POST \
  -f body='サマリー本文' \
  -f event='APPROVE or REQUEST_CHANGES or COMMENT' \
  -f 'comments[][path]=ファイルパス' \
  -f 'comments[][line]=行番号' \
  -f 'comments[][side]=RIGHT' \
  -f 'comments[][body]=コメント本文'
```

コメント数が多い場合はJSONファイルに書き出して `--input` で渡す:

```bash
gh api repos/{owner}/{repo}/pulls/{PR番号}/reviews --method POST --input review.json
```

```json
{
  "body": "サマリー本文",
  "event": "COMMENT",
  "comments": [
    {
      "path": "src/services/order.ts",
      "line": 42,
      "side": "RIGHT",
      "body": "🚨 **正確性**\n\n`user` が null の場合に..."
    }
  ]
}
```

インラインコメントの形式:

```
🚨 **正確性**

`user` が null の場合に NullPointerException が発生する可能性があります。

```suggestion
if (user != null) {
    ...
}
```
```

#### サマリー本文の形式

```markdown
## コードレビュー

### 概要
[変更内容の要約と全体評価を1-2文で]

### 変更ファイル

| ファイル | 変更概要 |
|---------|---------|
| `path/to/file` | [ロジック変更の詳細、影響範囲、不足している処理、付随する変更などを具体的に記述] |

変更概要の記述例:
- `app/Models/Order.php`: transaction_categoryの判定ロジックをタイプベース（MIDPRICE→LM、それ以外→YG）から受渡期日ベース（当日→TG、それ以外→YG）に変更。value_date_fromがnullの場合の防御的チェックが不足。スタイル修正（import順序、スペース）も含む。
- `tests/Unit/Models/OrderTest.php`: 新規テストファイル。銀行予約（一般・仲値・FXGO）における受渡期日に基づくtransaction_category判定（TG/YG）の5つのテストケースを追加。Carbon::setTestNowを使用した適切な日付制御。

### Confidence Score: N/5

[スコアの根拠を記述。減点要因がある場合は具体的なリスク・懸念事項をファイル名・行番号付きで説明する]

例:
> Confidence Score: 4/5
> ビジネスロジックの変更は明確で、テストカバレッジも良好。軽微なnullハンドリングの懸念あり。
> 変更は単一箇所のロジック修正に限定されており、テストが5パターンをカバーしている。ただし、value_date_fromがnullの場合にCarbon::parse(null)が現在日時を返す挙動により、誤ったTG判定が生じる可能性がある点をスコアに反映。フォームバリデーションで必須チェックが存在するため、実運用での発生リスクは低い。
> app/Models/Order.php の197行目、value_date_from のnullチェックの追加を推奨。

**スコア基準:**
- **5/5**: 問題なし。自信を持ってAPPROVE
- **4/5**: 軽微な懸念はあるが実運用リスクは低い
- **3/5**: 動作はするが改善すべき点が複数ある
- **2/5**: 潜在的なバグやセキュリティリスクあり
- **1/5**: 重大な問題があり、大幅な修正が必要

**主な減点要因:** ビジネスロジック・API・DB操作の変更にテストが伴わない場合は最低でも1点減点する

### 指摘サマリー
- 🚨 Must-fix: N件
- 💡 Suggestion: N件
- 🔍 Nit: N件

### 良い点
- [PRで良いと思った点]

[関連Issueがある場合]
### Issue要件との整合性
- [要件充足度の評価]

---
🤖 このレビューは Claude Code (`/code-review`) によって生成されました
```

`--self-review` 指定時はフッターを以下に変更する:

```
🤖 このレビューは Claude Code (`/code-review --self-review`) によって生成されました（セルフレビュー済み）
```

#### エラー時の対応

- `line` は差分に含まれる行のみ指定可能。差分外のコードへの指摘はサマリーに記載する
- 422エラー時は問題のコメントを除外し、サマリーに移動して再試行する

### 9. 完了報告

レビュー結果サマリーとGitHub上のレビューURLをユーザーに表示する。

`--self-review` が有効な場合は、セルフレビュー結果サマリー（除外・降格・統合・行番号修正の件数、Before/After比較）も合わせて表示する。
