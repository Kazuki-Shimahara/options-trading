# 要望調査 & Jiraチケット作成ワークフロー

要望・バグ報告を受け取り、コードベースを調査して、修正方針を含む構造化されたJiraチケットを作成する。

## 前提

- 引数 `$ARGUMENTS` に要望の概要テキストが渡される
- 引数がない場合はユーザーに要望内容を確認する
- Jira CLI が導入されていない場合はインストール手順を案内する

## 手順

### 1. Jira CLI の確認

```bash
jira version 2>/dev/null || echo "NOT_INSTALLED"
```

インストールされていない場合は以下を案内して停止する:
- macOS: `brew install ankitpokhrel/jira-cli/jira-cli`
- その他: https://github.com/ankitpokhrel/jira-cli#installation

初期設定が未実施の場合:
```bash
jira init
```

### 2. 要望の理解

引数から要望の本質を理解する。不明な点があればユーザーに質問する。

### 3. Jiraプロジェクト情報の確認

ユーザーに以下を確認する（CLAUDE.md に記載があればそちらを優先）:

- **プロジェクトキー**（例: `PROJ`）
- **課題タイプ**（例: `Task`, `Bug`, `Story`）
- **親チケット**（エピックや親ストーリーのキー。例: `PROJ-100`。不要なら省略）
- **Sprint**（現在のSprintに追加するか。不要なら省略）
- **担当者**（アサインするユーザー。不要なら省略）

### 4. コードベース調査

要望に関連するコードを徹底的に調査する。

#### 4a. 関連ファイルの特定
- Grep / Glob ツールで関連するキーワード・ファイルを検索する
- CLAUDE.md にアーキテクチャやディレクトリ構成の記載があれば参照し、調査の起点にする
- ルーティング定義から該当エンドポイントを辿り、コントローラー → サービス → モデルのデータフローを追跡する

#### 4b. 現状のコードの理解
- 該当機能のデータフローを追跡する
- 関連するバリデーション、計算ロジック、表示ロジックを確認する
- 同じ機能に複数のバリエーション（ロール別、画面別等）がある場合は全て確認する

#### 4c. 問題箇所の特定
- バグの場合: 原因となるコードの箇所を特定し、ファイルパスと行番号をメモする
- 機能追加の場合: 変更が必要な箇所を全て洗い出す

### 5. 修正方針の策定

調査結果をもとに修正方針を立てる:
- 修正が必要なファイルとその変更内容
- 修正の順序（依存関係がある場合）
- テストで確認すべきポイント
- 影響範囲（他機能への副作用がないか）

### 6. Jiraチケット作成

`jira issue create` でチケットを作成する:

```bash
jira issue create \
  --project "<プロジェクトキー>" \
  --type "<課題タイプ>" \
  --summary "<タイトル>" \
  --body "$(cat <<'EOF'
## 概要

[要望・バグの説明。何が問題で、どうあるべきかを1-2段落で記述]

## 現状

[現在のコードの該当箇所を示す。ファイルパス・行番号・コードスニペットを含める]

```
// ファイルパス:行番号
該当コードのスニペット
```

## 修正方針

[番号付きリストで具体的な修正手順を記述]

1. ...
2. ...
3. ...

## 対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `ファイルパス` | 変更の概要 |

## テスト観点

- [テストで確認すべきポイントをリスト化]

## 備考

[スコープ外の事項、注意点、質問事項があれば記載]
EOF
)"
```

### 7. 親チケットの紐づけ（指定がある場合）

```bash
jira issue link <作成したチケットキー> <親チケットキー> "is child of"
```

### 8. Sprintへの追加（指定がある場合）

```bash
# 現在のアクティブSprintを確認
jira sprint list --project "<プロジェクトキー>" --state active

# Sprintに追加
jira sprint add <Sprint ID> <作成したチケットキー>
```

### 9. 担当者のアサイン（指定がある場合）

```bash
jira issue assign <作成したチケットキー> <担当者>
```

### 10. 完了報告

作成したチケットのキーとURLをユーザーに表示し、調査で判明した主要な発見と修正方針の概要を報告する。
